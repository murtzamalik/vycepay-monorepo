# VycePay — Communication and Integration

## 1. Mobile App ↔ BFF

- **Protocol:** HTTP/HTTPS, REST, JSON.
- **Base URL:** Production: deployed BFF URL; dev: `http://localhost:8080` (Docker host often maps 9080→8080).
- **Auth:**  
  - Public: `POST /api/v1/auth/register`, `POST /api/v1/auth/login`, `POST /api/v1/auth/verify-otp`.  
  - All other `/api/v1/**`: header `Authorization: Bearer <JWT>`. Missing/invalid token → 401.
- **Customer context:** BFF does **not** expect `X-Customer-Id` from the client. It derives customer from JWT (externalId) and sets `X-Customer-Id` when proxying to backends.
- **Important headers:**  
  - `Idempotency-Key`: Required for POST /transactions/send; optional for POST /transactions/deposit/mpesa.  
  - `Content-Type: application/json` for requests with body.

---

## 2. BFF ↔ Backend Services

- **Mechanism:** BFF is a **reverse proxy**. Request path under `/api/v1/` is forwarded as-is; first segment selects backend:
  - `auth` → auth service (e.g. http://127.0.0.1:8082)
  - `kyc` → kyc service (8083)
  - `wallets` → wallet service (8084)
  - `transactions` → transaction service (8085)
  - `activity` → activity service (8086)
- **Headers:** BFF copies incoming headers (except Host/Connection/Content-Length) and **adds** `X-Customer-Id` (from JWT) for authenticated requests.
- **Failure:** If backend is unreachable or returns 5xx, BFF can return 502 with body `{ "code": "BAD_GATEWAY", "message": "..." }`. Unknown path prefix → 404.

---

## 3. Backend Services ↔ Choice Bank (BaaS)

- **Who calls:** KYC service (onboarding, sendOtp, resendOtp, confirmOperation, getOnboardingStatus); Transaction service (applyForTransfer, depositFromMpesa, getBankCodes, sendOtp, resendOtp, confirmOperation, history).
- **How:** Shared **Choice Bank client** in `vycepay-common`: `ChoiceBankClient`, `ChoiceBankRequestFactory`, `ChoiceBankSignatureUtil` (BaaS signing). Services use `BankingProviderPort` implemented by `ChoiceBankApiAdapter` which uses `ChoiceBankClient`.
- **Request format:**  
  - Base URL configurable (e.g. `https://baas-pilot.choicebankapi.com`).  
  - Each request: JSON body with `requestId`, `sender`, `locale`, `timestamp`, `salt`, `signature`, `params`.  
  - **Signature:** BaaS style: flatten params (including `params={}` when empty), sort keys, build string `key=value&...`, sign (SHA-256 with private key), include in body.  
  - **requestId:** Format `{senderId}{UUID without hyphens}` (e.g. VYCEIN + 32 hex chars). Generated by `RequestIdGenerator.generate(senderId)`.
- **Config:** `CHOICE_BANK_BASE_URL`, `CHOICE_BANK_SENDER_ID`, `CHOICE_BANK_PRIVATE_KEY` (or equivalent in application.yml). Same JWT secret as auth for token validation where needed.
- **Resilience:** Circuit breaker and retry (Resilience4j) can be configured on Choice client.

---

## 4. Choice Bank → VycePay (Callbacks)

- **Endpoint:** `POST /api/v1/choice-bank/callback` on **callback service** (port 8081). Not exposed via BFF to mobile; must be reachable by Choice Bank (public URL in production).
- **Body:** Raw JSON. Top-level fields include `requestId`, `notificationType`, `params` (nested payload).
- **Response:** HTTP 200 with body `"ok"` so Choice considers delivery success. Processing is **asynchronous** after persist.
- **Idempotency:** Callbacks with same (requestId, notificationType) are stored once; duplicate requests return existing record; processing runs once per unique callback.
- **Optional:** Callback signature verification (configurable); if enabled and verification fails, request is rejected (no persist/process) but implementation may still return 200 to avoid retries—see callback service config.

---

## 5. Services ↔ Database

- **DB:** Single MySQL (or MariaDB) instance; one schema (e.g. `vycepay`).
- **Access:** Each service uses Spring Data JPA; same JDBC URL, user, password. Configured via `SPRING_DATASOURCE_URL`, `DB_USERNAME`, `DB_PASSWORD` (or yaml).
- **Migrations:** Flyway in `vycepay-database`; single migration set; all services point to same DB so schema is shared.
- **Tables:** customer, otp_verification, kyc_verification, wallet, transaction, choice_bank_callback, activity_log. No direct service-to-service HTTP for normal flows; coordination via DB and callbacks.

---

## 6. Service-to-Service (None for App Flows)

- Backends do **not** call each other over HTTP for standard user flows.  
- BFF is the only caller of auth, kyc, wallets, transactions, activity.  
- Callback service only receives from Choice Bank and writes to DB; other services read the same DB.  
- So “communication” between auth, KYC, wallet, transaction is via **shared database** and **callback-driven updates** (callback service writing to kyc_verification, wallet, transaction).

---

## 7. Error Propagation

- **BFF:** 401 for missing/invalid JWT. 404 for unknown path prefix. 502 when backend call fails. Body shape: `{ "code", "message" }`.
- **Backends:** Return standard HTTP status and JSON error envelope (`code`, `message`, optional `requestId`, `details`). BFF forwards body from backend on 4xx/5xx (or substitutes 502 body on connection failure).
- **Choice Bank errors:** Returned in backend response (e.g. code 12004 “Invalid signature”); backend may map to BAD_GATEWAY or pass through. Mobile should show message to user and use code for logic (e.g. redirect to KYC).
